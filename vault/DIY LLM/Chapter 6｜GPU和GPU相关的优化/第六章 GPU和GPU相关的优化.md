# 第六章 GPU 和 GPU 相关的优化

> 来源：[Datawhale diy-llm 第六章：GPU 和 GPU 相关的优化](https://github.com/datawhalechina/diy-llm/blob/main/docs/chapter6/chapter6_%E7%AC%AC%E5%85%AD%E7%AB%A0GPU%E5%92%8CGPU%E7%9B%B8%E5%85%B3%E7%9A%84%E4%BC%98%E5%8C%96.md)

## 0. 一句话总览

这一章的核心是：**GPU 为什么适合大模型、GPU 内部怎么执行、显存为什么经常成为瓶颈，以及 FlashAttention / PageAttention 这类优化到底在优化什么。**

可以把整章理解成一条线：

```text
GPU 起源
  -> CPU vs GPU
  -> SM / Warp / Block / Thread 执行模型
  -> GPU 分层内存
  -> 计算越来越快，内存越来越跟不上
  -> 性能优化：少访存、多并行、低精度、算子融合、重计算、Tiling
  -> FlashAttention：让 Attention 少搬数据
  -> PageAttention：让 KV Cache 少浪费显存
  -> 国产 GPU / NPU 生态
```

## 1. GPU 为什么适合 AI

CPU 和 GPU 没有绝对优劣，设计目标不同。

| 对比 | CPU | GPU |
|---|---|---|
| 优化目标 | 低延迟，单个任务快 | 高吞吐，海量任务一起快 |
| 擅长 | 复杂逻辑、分支判断 | 大量重复的乘加计算 |
| 核心数量 | 少但强 | 多但相对简单 |
| AI 适配性 | 不适合大规模矩阵乘 | 非常适合矩阵乘 |

大模型训练和推理中最重的计算是矩阵乘法，而矩阵乘法的本质就是海量乘法和加法，所以 GPU 特别适合。

关键记忆：

```text
CPU：快快处理一个复杂任务
GPU：一起处理很多简单任务
```

## 2. GPU 的执行模型

记住四个概念：

| 名词 | 含义 | 类比 |
|---|---|---|
| SM | Streaming Multiprocessor，GPU 的基本计算单元 | 一个小工厂 |
| Block | 线程块，通常分配给一个 SM | 一个施工队 |
| Warp | 线程束，NVIDIA 通常 32 个线程 | 一辆车，大家一起行动 |
| Thread | 最小执行单元 | 工人 |

### 2.1 SIMT

GPU 使用 SIMT：Single Instruction, Multiple Threads。

意思是：**同一条指令，多个线程处理不同数据。**

这非常适合矩阵、向量这类规则计算。

### 2.2 Warp Divergence

同一个 Warp 里的线程最好走同一条分支。

如果一部分线程走 `if`，另一部分走 `else`，GPU 会把两个分支都执行一遍，只是用 mask 屏蔽暂时不该执行的线程。这样就会从并行变成近似串行，性能下降。

```text
优化原则：同一个 Warp 内尽量不要发生分支发散。
```

## 3. GPU 的内存模型

GPU 优化绕不开一句话：

```text
离 SM 越近，越快，但越小越贵。
```

| 层级 | 速度 | 容量 | 特点 |
|---|---|---|---|
| 寄存器 | 最快 | 很小 | 线程私有 |
| 共享内存 / L1 | 很快 | 小 | Block 内共享，可手动控制 |
| L2 Cache | 中等 | 较大 | 所有 SM 共享，硬件管理 |
| 全局显存 HBM | 慢 | 最大 | 存权重、激活、KV Cache 等 |

这章反复强调的核心矛盾：

```text
GPU 算得越来越快，但显存/内存带宽增长没那么快。
所以瓶颈经常不是“算不动”，而是“数据搬太慢”。
```

## 4. TPU 简单理解

TPU 是 Google 做的 AI 专用芯片。

GPU 是从图形计算转向 AI，TPU 更像是从出生就为张量/矩阵运算设计。

| 对比 | GPU | TPU |
|---|---|---|
| 定位 | 通用并行计算，图形 + AI + 科学计算 | AI 专用加速 |
| 重点 | 灵活、生态强 | 矩阵乘法极致优化 |
| 编程生态 | CUDA 等 | TensorFlow / XLA 等 |
| 适用性 | 更通用 | 更专用 |

## 5. 性能扩展趋势

现代 GPU 的计算能力提升很快，但内存带宽提升没那么快。

这就带来所谓的 **Memory Wall 内存墙**。

屋顶线模型可以帮助理解：

- 横轴：操作强度，也就是每搬一个字节能做多少计算。
- 纵轴：吞吐量，也就是每秒能做多少计算。
- 左侧通常是内存受限：数据搬太慢。
- 右侧通常是计算受限：计算单元终于吃饱了。

所以优化方向是：

```text
减少不必要的全局内存访问，
提高每次数据搬运后能做的计算量，
让计算单元尽量一直忙。
```

## 6. 主要性能优化技术

这一节最重要，可以总结成一句：

```text
尽量让计算单元一直忙，尽量少访问慢速全局内存。
```

| 技术 | 核心思想 |
|---|---|
| 避免分支发散 | 同一个 Warp 内尽量走同一路径 |
| 低精度 | FP16/BF16/INT8/FP8 数据更小，算得更快、搬得更少 |
| 算子融合 | 多个操作合成一个 Kernel，避免中间结果反复写回显存 |
| 重计算 | 不存某些中间激活，反向传播时重新算，用计算换显存 |
| 内存合并 | 让线程访问连续地址，提高显存带宽利用率 |
| Tiling 分块 | 把大矩阵切块，先搬到共享内存里反复用，减少全局显存访问 |

### 6.1 低精度

低精度能提速，主要有三个原因：

1. 数据位数更少，显存占用更低。
2. 数据搬运量更少，同样带宽能搬更多数据。
3. Tensor Core 对低精度矩阵乘有专门加速。

常见精度：

| 精度 | 常见场景 |
|---|---|
| FP32 | 传统训练，高精度 |
| FP16 | 混合精度训练和推理 |
| BF16 | 大模型训练常用，动态范围更友好 |
| TF32 | A100 之后常见，兼顾速度和范围 |
| INT8 / INT4 | 量化推理 |
| FP8 | Hopper / Blackwell 等新架构重点支持 |

### 6.2 算子融合

普通写法可能是：

```text
读 x -> 算 sin -> 写回
读 sin 结果 -> 平方 -> 写回
读另一个中间结果 -> 再算 -> 写回
```

算子融合的思路是：

```text
读一次 x -> 在一个 Kernel 里连续算完 -> 最后写回一次
```

本质是减少中间结果在显存里的读写。

### 6.3 重计算

训练时反向传播需要前向的中间激活。

普通做法：前向时把激活都存下来，反向时读出来。

重计算做法：前向时少存一点，反向时重新算需要的中间结果。

本质：

```text
用额外计算换更少显存访问。
```

适合场景：

- 显存很紧。
- 计算单元还有空闲。
- 重算成本小于读写成本。

### 6.4 内存合并

GPU 访问全局内存时，如果同一个 Warp 内的线程访问连续地址，硬件可以把它们合并成更少的内存事务。

好模式：

```text
thread0 -> A[0]
thread1 -> A[1]
thread2 -> A[2]
thread3 -> A[3]
```

差模式：

```text
thread0 -> A[0]
thread1 -> A[1000]
thread2 -> A[2000]
thread3 -> A[3000]
```

前者能更好利用显存带宽。

### 6.5 Tiling 分块

矩阵乘法里，很多元素会被重复使用。

如果每次都从全局显存读取，会很慢。Tiling 的做法是：

```text
先把一小块矩阵从全局显存搬到共享内存
在共享内存中反复使用
算完再处理下一块
```

这样可以显著减少全局显存访问。

## 7. FlashAttention

普通 Attention 的问题：

```text
QK^T 会产生一个 N x N 的注意力矩阵。
序列越长，这个矩阵越恐怖，显存访问和中间结果都很重。
```

FlashAttention 的核心：

```text
不显式保存完整 attention 矩阵。
把 Q / K / V 分块，边算边做 softmax，尽量把数据留在 SRAM / 共享内存里。
```

三个版本的重点：

| 版本 | 解决什么 |
|---|---|
| FlashAttention V1 | 解决 IO 瓶颈，减少 HBM 访问 |
| FlashAttention V2 | 提高并行度和 Tensor Core 利用率 |
| FlashAttention V3 | 利用 H100 的异步执行、WGMMA、FP8，让计算和搬数据重叠 |

最好记的版本总结：

```text
V1：少搬数据
V2：更会并行
V3：边搬边算，让 SM 少等
```

## 8. PageAttention

PageAttention 主要解决推理时 KV Cache 的显存浪费。

传统 KV Cache：

```text
每个请求提前分配一大段连续显存。
但生成长度不可预测，很多空间会浪费。
```

PageAttention：

```text
把 KV Cache 切成固定大小的 page / block。
逻辑上连续，物理上可以不连续。
通过页表把 token 映射到真实物理块。
```

可以理解成：

```text
虚拟地址 -> 页表 -> 物理地址
```

更具体：

```text
请求 ID + Token 偏移量
  -> 逻辑块索引
  -> Block Table 检索
  -> 物理内存基址
  -> 偏移读取
```

如果每个 page 有 16 个 token，一个请求实际生成 300 个 token，就只需要 19 个 page，也就是 304 个 token 容量，只浪费 4 个 token。

如果传统方式按最大长度 2048 预分配，就会浪费 1748 个 token 容量。

## 9. FlashAttention vs PageAttention

| 对比 | FlashAttention | PageAttention |
|---|---|---|
| 优化层级 | 算子级 | 系统级 |
| 解决问题 | Attention 算得更快 | KV Cache 存得更省 |
| 主要场景 | 训练 + 推理 | 主要是推理 |
| 核心关键词 | IO-aware、tiling、online softmax | KV Cache、分页、block table |

一句话：

```text
FlashAttention 管“怎么算得快”，PageAttention 管“怎么存得省”。
```

实际推理框架里二者经常协同使用，比如 vLLM 这类系统中会同时关心 attention kernel 的速度和 KV Cache 的显存管理。

## 10. 国产 GPU / NPU 部分

这一节主要介绍国产 AI 芯片生态，包括：

- 沐曦

- 天数智芯

- 阿里平头哥

重点不是背每个参数，而是理解国产 AI 芯片竞争主要看这些维度：

```text
算力指标
显存容量 / 带宽
卡间互联能力
软件生态

实际大模型落地案例
```

其中华为昇腾生态相对成熟，软件栈是 CANN，类似 NVIDIA CUDA 在昇腾体系里的位置。

## 11. 最该记住的 10 个点

1. CPU 优化低延迟，GPU 优化高吞吐。
2. 大模型适合 GPU，因为核心计算是大规模矩阵乘。
3. SM 是 GPU 的基本计算单元。
4. Warp 是 GPU 调度的基本单位，NVIDIA 通常 32 线程。
5. Warp Divergence 会导致分支串行化，拖慢性能。
6. GPU 性能瓶颈常常是显存访问，不是计算本身。
7. 低精度能减少数据量，也能用 Tensor Core 加速。
8. 算子融合、重计算、Tiling 都是在减少慢速显存访问。
9. FlashAttention 优化 Attention 的 IO 和并行执行。
10. PageAttention 优化推理时 KV Cache 的显存管理。
